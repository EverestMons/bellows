- **Added 2026-05-21:** Step-counter loop after precondition-failure verdict. When Bellows pauses on a precondition gate (worktree creation failure) labeled `step-N` and the Planner verdict-continues, Bellows re-dispatches step N rather than advancing to N+1. If step N's work was already shipped to origin from a prior run (the parallel-SHA pattern means origin can have the commit while local doesn't), the agent re-runs in a fresh worktree forked from origin tip → edits are no-ops → resulting commit is empty → teardown's cherry-pick fails empty → gate_failure → re-pause on the same step-N. Loops indefinitely until Planner stops with `verdict: stop`. Reproduction: today's `executable-fuel-continuation-inference-ui-2026-05-21` plan had this exact loop on Step 2 (Step 1 + Step 2 work was on origin from the first run; the 15:38 daemon restart triggered re-claim, the 15:46 verdict-continue on a worktree-creation step-1 request advanced to Step 2 dispatch, Bellows re-ran Step 2 fresh from origin tip, agent no-op'd, teardown failed empty, re-pause, re-dispatch with the same outcome — three iterations before Planner verdict: stop). Two likely root causes to investigate: (1) `_consume_verdicts` interprets continue-on-precondition-failure as "step N done, advance to N+1" without checking whether step N actually produced work, even though the verdict request's `Pause Reason Code: gate_failure` on `worktree_creation` is distinguishable from a header_pause; (2) step counter is computed from the verdict request's `Step` field rather than from internal daemon state, so a precondition-failure labeled `step-1` propagates as "step 1 resolved, start step 2." Resolution: diagnostic to characterize the dispatch-after-verdict logic in `bellows.py`, then either (a) detect "precondition gate failure" Pause Reason Codes and resume from the same step rather than advancing, or (b) make `step` field in verdict requests reflect the next-step-to-dispatch rather than the most-recently-paused-step. **Operational mitigation today:** Planner verdict: stop on the second teardown-empty in a row for the same step, then author a follow-up plan covering only the remaining steps. Recovery cost is ~10 minutes per occurrence (stop verdict + new plan + worktree cleanup + git reset). Bellows session log evidence: 15:38:18 → 15:39:50 → 15:50:27 → 15:52:48 loop visible in `logs/terminal/bellows-2026-05-21.log`. Reference: LESSONS 2026-05-27 parallel-SHA pattern (related but distinct — that's about teardown semantics, this is about post-verdict step advancement).
