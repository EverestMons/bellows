882:Bellows-side fix-plans that modify plan-text parsers — `_extract_step_text`, `_gate_is_qa_step`, `extract_total_steps`, `_extract_step_text_from_plan`, `strip_fenced_code_blocks` — will trip the bug they fix during their own execution because the running daemon executes the pre-fix code through the entire fix-plan's lifecycle. The failure typically manifests as a gate trip (`rule_20_self_check` or `deposit_exists`) on the QA step, where the parser producing the step text or gate input is the very parser the fix-plan corrects. The CEO override at this gate failure is correct: the fix is verifiably good in the committed code, and the gate failure is an artifact of the stale running process, not a defect in the fix. The Planner pre-documents this expected override pattern in the plan header so the CEO knows to anticipate the QA-step gate failure and can authorize the override without investigation. After the daemon restarts with the fixed code, a bait-laden canary plan (see LESSONS.md 2026-05-11 entry on canary verification) should be deposited to confirm the fix loaded correctly. The structural inevitability of this failure mode — fix-plan correctness depending on code that does not load until after the plan completes — means the gate trip is a known cost of parser fix-plans, not a signal to halt.
