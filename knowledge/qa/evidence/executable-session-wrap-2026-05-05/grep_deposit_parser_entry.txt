11:- 2026-05-05: deposit-parser captures backtick-wrapped paths from step prose with trailing description fragments as literal deposit paths. Observed during executable-backlog-1-close-and-lessons-2026-05-05 Step 1: deposit_exists tripped on three "missing" paths formatted as path-then-backtick-then-em-dash-description (e.g., bellows/knowledge/BACKLOG.md backtick em-dash BACKLOG #1 closed, ...). All three deposit files actually existed and were correctly modified. Root cause: parser keyword-scans step prose for path-like strings instead of strictly using the declared **Deposits:** block; when prose mentions a deposit path inline as part of agent instructions (a common authoring pattern), parser concatenates path with following description text and reports the resulting nonsense string as missing. Same failure-class family as BACKLOG #6 (closed 2026-04-19 via gates.py block-scoping for plans WITH a **Deposits:** block) — but this plan HAD a properly-formatted **Deposits:** block, so the gate fell back to prose scanning despite the block being present, OR scanned both and merged results. Fix shape candidates: (1) when a **Deposits:** block is present, scope deposit_exists ONLY to paths declared in the block; ignore prose entirely; (2) tighten the parser regex to reject paths immediately followed by a backtick character (a backtick after a path-shape means it was a backtick-wrapped reference inside prose, not a declaration); (3) require deposits to be specified ONLY in the block (governance change, no parser change) — but this conflicts with the readable convention of citing deposit paths in agent instructions. Recommendation: option (1) — declared-block-takes-precedence is the structural fix; matches BACKLOG #6's spirit. Workaround until shipped: Planner reads the prose-scanning false-positive verdict, applies Rule 22 to the actual deposits, overrides via continue verdict (~30s of friction per plan that hits this pattern).
