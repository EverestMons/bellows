# Bellows — Remove pre-scan `processed-` prefix rename (Shape 1c, P0 loop fix)
**Date:** 2026-05-24 | **Tier:** Small | **Dispatch Mode:** bellows | **Test Scope:** full | **Execution:** Step 1 (DEV) → Step 2 (QA) | **pause_for_verdict:** after_step_1 | **auto_close:** false

## Context

Per the 2026-05-24 diagnostic findings at `bellows/knowledge/architecture/processed-prefix-reconsumption-and-rename-skip-2026-05-24.md`, the pre-scan rename block at `bellows.py:1129-1162` causes a P0 infinite loop on multi-step plans that pause for verdict between steps. The mechanism (Section B of the diagnostic):

1. Step 1's verdict is consumed normally; `verdict-*-step-1.md` is renamed to `processed-verdict-*-step-1.md` at `bellows.py:1291`.
2. Step 2 runs, completes, pauses → plan is `verdict-pending-*`.
3. Pre-scan finds the on-disk `processed-verdict-*-step-1.md`, orphan-guard sees a `verdict-pending-*` plan with matching slug (no step-number check), collision-guard sees no `verdict-*-step-1.md` exists, **renames `processed-verdict-*-step-1.md` back to `verdict-*-step-1.md`.**
4. Main loop consumes the renamed file as a fresh step-1 continue verdict, advances the plan to re-run step 2.
5. Cycle repeats indefinitely. No natural termination mechanism exists.

Reproduction: `executable-stuck-state-color-override-2026-05-22` looped 14+ times over 3 hours including a 600s inactivity-timeout kill before manual intervention.

**Why Shape 1c (full removal) rather than Shape 1a (step-number guard) or Shape 1b (mtime guard):** PLANNER_TEMPLATE v4.49 Rule 25 shipped 2026-05-21 prohibits the `processed-` prefix at write time. The governance rule is the authoritative fix for the original "Planner writes `processed-` by mistake" failure mode. Zero violations of the rule have been observed since it shipped. The pre-scan was a belt-and-suspenders safety net that has now created a new bug class worse than the one it prevented. Removing it eliminates the loop class permanently with negative LOC.

**What gets removed:**
- Pre-scan block at `bellows.py:1129-1162` (~34 lines)
- `_prescan_orphan_logged: set` module-level state at `bellows.py:33` (orphan-guard helper, becomes dead code)
- The four pre-scan regression tests in `tests/test_consume_verdicts.py` (test names from diagnostic Section G Q19):
  - `test_pre_scan_renames_processed_verdict_to_canonical`
  - `test_pre_scan_collision_guard_does_not_overwrite`
  - `test_pre_scan_skips_rename_when_no_paired_plan`
  - `test_pre_scan_renames_when_verdict_pending_plan_exists`

**What stays:**
- The main verdict consumption loop at `bellows.py:1164+` (unchanged — handles all canonical `verdict-*` files correctly)
- The post-consumption rename at `bellows.py:1291` that produces `processed-verdict-*` files in the first place (unchanged — this is the legitimate Bellows-side marker that a verdict has been consumed)
- The no-match WARN at `bellows.py:1317` (unchanged — surfaces any future write-time `processed-` violation for manual triage)
- Rule 25 governance language in `PLANNER_TEMPLATE.md` v4.49 (unchanged — remains the authoritative fix)

**Acknowledged scope:** This executable removes work shipped in two prior plans — commit `3c9344f` (the 2026-05-21 pre-scan rename fix) and commit `d1855ba` (the 2026-05-22 orphan guard, `executable-pre-scan-orphan-guard-2026-05-22`). Both shipped legitimate fixes for their stated bugs, but they targeted a problem that governance now prevents. The pre-scan's removal is a deliberate hardening choice, not a regression. The governance rule (Rule 25) is the source of truth for the no-`processed-`-at-write-time invariant.

**Daemon-restart note:** This fix touches `bellows.py`, which Bellows does not hot-reload (Restart Discipline). The running daemon will continue executing pre-fix code through this plan's lifecycle. Expected behavior: gates pass on Step 1 and Step 2 because the pre-scan only fires on plans whose step-1 verdict has already been consumed and is sitting in `resolved/`. This plan's own step 1 verdict will be in `resolved/` after step 1 completes, but step 2 is the QA step on the same plan, so the loop trigger won't reach a re-dispatch (this plan only has 2 steps, and step 2 is QA, not a DEV re-run). After plan close + daemon restart, the pre-scan removal becomes effective for all future plans.
