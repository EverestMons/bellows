**Edit 1 — Explicit no-push instruction.** The 2026-05-21 teardown-git-operations diagnostic (`bellows/knowledge/decisions/Done/diagnostic-teardown-git-operations-mapping-2026-05-21.md`) identified push instructions in plan prose as the root cause of parallel-SHA divergence: when agents run `git push` from inside a Bellows worktree, the commit lands on origin but the worktree branch tip diverges from local main. The 2026-05-21 verdict-enrichment plan (`Done/executable-bellows-verdict-enrichment-2026-05-21.md`) validated the mitigation empirically — 4 agent commits, zero parallel-SHA divergence — by deliberately omitting push from step prose. Current Rule 23(c) text says "feedback append → final commit" with no push verb, but it does NOT explicitly forbid agent-side push, so agents pattern-match push from older plans and specialist file boilerplate. Fix is additive: add one sentence to Rule 23(c) and Rule 8's mirror paragraph explicitly forbidding agents from running `git push` during step execution; the Planner handles push at session-wrap.

**Pre-write contradiction scan (per Rule 22):** Completed 2026-05-21. Searched for all `git push` references in PLANNER_TEMPLATE.md v4.46. Findings: (a) Rule 31 (submodule pointer bump) contains `git --no-pager push origin main` — this is a Planner-side session-wrap activity, NOT an agent-step instruction; removing it would break submodule pointer propagation; remains unchanged. (b) Procedure 3 (`git filter-repo` post-execution) contains `git push origin main --force-with-lease` — this is an operator-side recovery procedure; remains unchanged. (c) No other `git push` references in the file. The new agent-side prohibition does not contradict either Planner-side or operator-side push activity.
