This means the failure mode is NOT "subprocess never started." The subprocess started, did partial file I/O, and then errored before completing the verification + commit. Then Bellows recorded `is_error=True` and `Blocked` status without any JSON log.

## Three suspect dimensions

The diagnostic should examine these three dimensions of difference between Plan 2b's failed dispatches and Plan 2c's successful one:

1. **Plan content / size.** Plan 2b is ~180 lines, larger than typical. Could be hitting a subprocess argument size limit, a bash command-line length limit, or a claude -p prompt size threshold.
2. **What the agent actually does.** Plan 2b uses a Python script approach (write a file via `with open()` then verify by re-reading). Plan 2c used direct str_replace edits via Filesystem MCP. The Python approach involves more tool-use turns and more state mutation. Could be hitting a per-step iteration limit or a tool-use budget.
3. **Bellows' subprocess construction.** Bellows builds the bootstrap prompt and invokes claude -p. The construction logic is in `bellows.py` and `runner.py`. There may be a code path that fails silently for certain plan shapes — the missing JSON log is the strongest signal of this (claude -p normally always outputs JSON when invoked with --output-format json).
