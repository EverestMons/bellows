# Bellows — scope_check Monorepo Topology Diagnostic
**Date:** 2026-04-28 | **Tier:** Small | **Test Scope:** targeted | **Execution:** Step 1 (Bellows Systems Analyst)
**Priority:** 5

## Context

BACKLOG #4 (`scope_check false-positive over too-wide git range`) reproduced cleanly on today's `executable-deposit-exists-prose-canary-2026-04-28` — both Step 1 and Step 2 tripped scope_check on `LESSONS.md` despite the agent never modifying it. Refined diagnosis from the canary's evidence: `bellows/` lacks its own `.git` directory and lives inside the governance-root repo at `/Users/marklehn/Desktop/GitHub/`. When `_capture_git_diff()` runs `git diff --stat` from `cwd=/Users/marklehn/Desktop/GitHub/bellows`, git walks up to the governance-root `.git` and reports the entire monorepo's diff, including unrelated files in sibling project directories. The existing `..`-component filter in `_parse_diff_stat` doesn't catch this because monorepo-relative paths are clean (no `..` components).

Proposed fix shape from the BACKLOG note: detect repo-root mismatch (`git rev-parse --show-toplevel` ≠ `project_path`) and scope the diff to the project subtree. But this proposal has unverified assumptions that must be resolved before any executable plan: the topology of all watched_projects (which are nested vs. standalone), the exact behavior of `git rev-parse --show-toplevel` from each, the parsing impact of path-scoping syntaxes on `_parse_diff_stat`'s output assumptions, the inventory of existing tests that may break, and whether any legitimate plan in history committed cross-repo files (false-negative risk).

This is a single-step diagnostic. SA investigates and reports findings. A separate executable plan will follow after CEO reviews findings.

Test Scope: targeted — investigation only, no code changes, no tests run.
