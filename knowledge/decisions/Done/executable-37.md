# Bellows — Store the Plan Document Path on plans (plan_doc_ref)
**Date:** 2026-06-13 | **Tier:** Medium | **Dispatch Mode:** bellows | **Execution:** Step 1 (DEV) → Step 2 (QA) | **pause_for_verdict:** always | **qa_steps:** 2 | **Test Scope:** full suite

## Context (Rule 27)
CEO observation 2026-06-13: the lifecycle DB has no authoritative pointer to the plan's source markdown — the document the daemon reads its instructions from. The only plan-file column, `plans.deposit_placeholder_name`, holds the transient pre-mint draft name (e.g. `executable-draft-235325.md`) and is never updated as the file moves through `in-progress-<type>-<id>.md` → `Done/<type>-<id>.md`. To locate the plan doc today, the Forge reporter GUESSES — `_resolve_plan_file()` reconstructs `<type>-<id>.md` and probes `Done/`, then `in-progress-`, then bare. CEO decision: make the plan document a first-class stored pointer. Placement decided: a single `plan_doc_ref` column on the shared `plans` table (NOT duplicated in each meta table — the doc is one per plan regardless of type; one column, one write path, updated by the daemon at each rename it already performs). This is faithful to the coverage-map principle "index the pointers, don't warehouse content" — a path is a pointer. Stored value = project-relative path (matches `deposits.declared_path` convention), reflecting the file's CURRENT location. Scope: bellows only (migration + writer + backfill). The Forge reader switching to prefer the stored ref over derivation is a deliberate follow-up (Step 2 files a FORWARD row), not in this plan — new rows being populated does not break the reader, which still resolves correctly by derivation.

Write sites mapped 2026-06-13 (re-verify by grep, do not trust line numbers): claim rename `shutil.move(plan_path, inprogress_path)` then `mark_plan_state(plan_id, "in_progress")` (~bellows.py:459-462); close move to `Done/` (~bellows.py:494, in the verdict-consume final-step branch); halt move to `halted-` (~bellows.py:418-419); central state writer `mark_plan_state` (~lifecycle.py:172). DAEMON RESTART REQUIRED after close (flag in receipt); the migration runs at the next daemon startup against the live `lifecycle.db`.
