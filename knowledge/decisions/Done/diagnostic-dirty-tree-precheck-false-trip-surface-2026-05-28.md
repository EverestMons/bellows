## Context

The `worktree_teardown_dirty_tree` pre-check shipped session 12 (commit `6252f8c`) added a `git status --porcelain` check immediately before the cherry-pick loop in `_teardown_worktree` (insertion at bellows.py:855 per the original surface diagnostic `worktree-teardown-dirty-tree-precheck-surface-2026-05-27.md`). The check raises `WorktreeTeardownError` whenever `git status --porcelain` returns ANY non-empty output, routing to a `gate_failure` pause with an evidence string leading `worktree_teardown_dirty_tree:`.

**The session-13 finding:** the check is net-negative for multi-step plans because it fires on Bellows's own untracked lifecycle artifacts, not real cherry-pick conflicts. Three trips were observed across two plans in session 13 (`diagnostic-rescan-miss-disposition-2026-05-28` and `executable-seen-discard-rejection-path-2026-05-28`) — every trip on accumulated lifecycle bookkeeping (`verdict-pending-*`, `processed-verdict-*`, `in-progress-*`, Done/ plans, `verdict-request-*`), zero on real source-tree changes that would have caused cherry-pick conflicts. On the `_seen` executable, the cascade compounded: teardown abort → stranded worktree → next-step `worktree_creation` precondition failure → precondition-retry resumes the SAME step → re-trips teardown → loop. Broken only by closing the executable Planner-direct (commit `3b9a997`).

The BACKLOG entry (Added 2026-05-28) proposes "narrow the dirty-tree check to ignore Bellows-managed paths." The prefix set in the entry is a starting point, not a verified enumeration — getting it wrong either re-introduces the cryptic cherry-pick conflict (under-filter: a real conflict is masked) or fails to fix the false-trip (over-filter or missed prefix).

**This diagnostic does NOT ship code.** Its job is to produce a verified filter-rule table that a follow-on executable plan can mechanize directly. The diagnostic must also disposition BACKLOG #2 (failed teardown strands worktree → blocks next step's worktree creation): is it a separate failure mode, or a downstream symptom of #1 that closes when the false-trip closes?
