## Context

BACKLOG #4: when a plan file is renamed from `verdict-pending-*` back to `executable-*` (manual recovery, or via Bellows-driven flow we don't fully understand yet), step state does not persist. Observed 2026-04-18: `executable-create-forge-developer-specialist-2026-04-18.md` had Step 1 complete and was renamed to pending for resume; on re-claim Bellows reported `plan has 2 steps, ▶ started` and dispatched Step 1 from scratch. The agent's defensive idempotency check caught it ($0.18 wasted). Code-change plans with non-idempotent effects (schema migrations, mutable state, non-idempotent commits) would corrupt state on re-run.

This diagnostic maps the current state machine. It does NOT propose a fix. The BACKLOG entry enumerates three fix-shape options (DB persistence, filename-metadata, idempotency declaration); choosing among them requires first knowing exactly what the current behavior is and where state lives. A separate Phase 2 plan will take these findings and produce a design comparison; Phase 3 will implement the chosen option.

This is investigation-only. Do not modify code. Single-step diagnostic per Rule 22.

Test Scope: none — pure investigation.
