## STEP 2 — BELLOWS_SYSTEMS_ANALYST (Per-Artifact Response Evaluation)

Read your specialist file and the Step 1 deposit at `bellows/knowledge/research/planner-authored-contract-validation-step-1-enumeration.md` first. **Part 3 — Three architectural responses evaluated per artifact.** For each artifact enumerated in Step 1, evaluate which of three responses is appropriate. **(A) Schema validator** — Bellows validates the Planner's file format on read, raises a specific error if malformed, posts a notification or logs to a visible location. Cost: ~20-40 LOC per validator + tests, ongoing schema maintenance. Pros: catches Planner errors immediately; allows Planner format to evolve without breaking consumers. Cons: doesn't eliminate the authorship surface; Planner still authors and can still get it wrong (just faster feedback). **(B) Writer helper** — Bellows ships a helper function the Planner calls instead of hand-writing the file. The helper is the single source of truth for the format. The Planner provides semantic inputs (e.g., plan path, step number, decision, rationale); the helper produces the bytes. Cost: ~30-60 LOC per helper + tests. Pros: makes the format-mismatch failure mode structurally impossible; format changes ship in one place. Cons: requires Planner-side discipline change (call helper, don't write file); the helper API is itself a contract the Planner must learn, though smaller than the file format. **(C) Observability-only** — accept the current parse behavior but add loud logging when a file in the watched directory does not match the parser's expectations. Cost: ~5-10 LOC per consumer. Pros: zero new contract surface; cheapest by far. Cons: failures still happen; observability only helps if the CEO sees the log; doesn't prevent recurrence. For each artifact, recommend one of A, B, or C with a one-paragraph justification grounded in the artifact's specific failure mode (from Step 1's Part 2), authoring frequency (high/medium/low), and known reproduction history. Where the response could reasonably be a combination (e.g., A+C: validator that logs loudly when it rejects), name the combination. Do NOT default to one response across all artifacts — evaluate each on its own merits. **Part 4 — Cross-cutting pattern observation.** After producing per-artifact recommendations, identify any patterns. If ≥75% of artifacts converge on B (helper), the right architectural move is a writer-helper-library approach (single module, multiple helpers, shared format-versioning). If recommendations split across two responses, the right move is per-artifact case-by-case. If they fragment across all three, the gap is larger than expected and the conversation needs to expand. Report the pattern observed with raw counts (e.g., "3 of 5 artifacts recommend B, 1 recommends A, 1 recommends C"). **Part 5 — Implementation cost summary and prioritization.** Produce a final table: artifact | recommended response | LOC estimate (sum if combination) | priority. Priority is **high** if the artifact has failed within the last 30 days AND its failure mode is silent-skip. **Medium** if it has failed within 30 days but is loud-failure (CEO already sees those), OR if no known failure but silent-skip mode (latent risk). **Low** if no known failure and loud-failure mode (low actual risk). Do NOT propose fix plans. Do NOT recommend execution order beyond the priority field. The Planner sequences based on CEO priorities after seeing the table. Deposit Step 2 findings to `bellows/knowledge/research/planner-authored-contract-validation-step-2-evaluation.md` with structure: Part 3 per-artifact A/B/C recommendation with justification, Part 4 cross-cutting pattern, Part 5 prioritized implementation summary table.
