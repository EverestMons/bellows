# Diagnostic — Intermediate Decision Detection Design Options

**Execution Map:** Step 1 (BELLOWS_SYSTEMS_ANALYST)

**Purpose:** The 2026-05-12 capture audit (`bellows/knowledge/research/output-receipt-capture-audit-2026-05-12.md`) established that the Output Receipt captures 2.7% of agent-narrated decisions in the `claude -p` NDJSON stream. The data exists on disk inside `raw_output`; it is not exposed to the Planner. Before designing a fix, we need to know which detection mechanism reliably separates substantive decisions (β-A and β-B class — agent self-corrects an empty evidence file; agent catches a logical error mid-implementation) from borderline-or-trivial procedural narration (β-C/D/E class — agent narrates which existing test to study; agent says "now I'll commit"). The audit used a 33-phrase grep list to *count* decisions but did not validate it as a *detector*. This diagnostic builds a hand-classified ground-truth set from the three worst-leak plans and tests four candidate detection mechanisms against it, reporting precision and recall for each. Output: a methodology recommendation, not a fix plan. The fix plan follows separately after CEO selects a detector.

## STEP 1 — BELLOWS_SYSTEMS_ANALYST

Read your specialist file and domain glossary first. Then read the prior audit findings at `bellows/knowledge/research/output-receipt-capture-audit-2026-05-12.md` (Part 7 representative β instances and worst-three-plans table specifically). **Part 1 — ground-truth construction.** Locate the per-step `step.json` artifacts for the three worst-leak plans identified in the prior audit: `deposit-exists-prose-canary-2026-04-28`, `bellows-worktree-impl-2026-05-03`, and `terminal-capture-2026-05-12`. For each plan, extract every `assistant` event's text content from the `raw_output` NDJSON stream — these are the intermediate narration blocks. Number them sequentially per plan-and-step (plan-A-step-1-block-1, plan-A-step-1-block-2, etc.). Report the total block count across all three plans. **Part 2 — hand classification.** For each intermediate narration block, assign exactly one of three labels based on the content of the block alone (no information from the agent's final result text, no information from the plan): **(S) substantive decision** — the agent describes detecting and resolving an unexpected condition (failed assumption, empty output, wrong path, logical error caught mid-work) OR makes a scope-changing choice not directly mandated by the plan (modifying a file not declared in deposits, expanding the change set, choosing a non-obvious implementation approach). The defining property: a Planner reading this block during Rule 22 would learn something not derivable from the deposit alone. **(B) borderline** — the agent narrates a research or methodology choice (which existing test to study, which file to read first, what to grep for) that affects HOW the work was done but not WHAT was delivered. A Planner would find this informative but not action-changing. **(T) trivial** — procedural narration ("now I'll commit", "let me run the test"), intent statements, status echoes, or content that paraphrases what the deposit already says. A Planner reading this would learn nothing. Apply labels conservatively — when ambiguous between S and B, pick B; when ambiguous between B and T, pick T. Bias toward T. Report the label distribution across the three plans. Deposit the labeled set to `bellows/knowledge/qa/evidence/intermediate-decision-detection-2026-05-12/labeled-blocks.jsonl` with one block per line: `{"plan": "...", "step": N, "block_idx": M, "text": "...", "label": "S|B|T"}`. **Part 3 — candidate detector implementation.** Implement and run four candidate detectors against the labeled set. Each detector takes the NDJSON stream as input and outputs a flagged subset of `assistant`-event text blocks. Detector A — phrase-grep (audit baseline): flag any block matching one or more of the 33 phrases from the prior audit's final extended list. Detector B — tool-failure-bracketed: flag any `assistant`-event text block that appears in the NDJSON stream within ±3 events of a `tool_use_result` event with non-zero exit code, non-empty stderr, or an `is_error: true` field. Detector C — write-tool-adjacent: flag any `assistant`-event text block within ±2 events of a `tool_use` event for write-class tools (Write, Edit, MultiEdit, Bash with redirect operators, Filesystem:write_file, Filesystem:edit_file, Filesystem:move_file, Desktop Commander:write_file, str_replace, create_file). Detector D — assistant-text-length percentile: flag any `assistant`-event text block whose character length is in the top 20% of all assistant-text-block lengths within that step. For each detector, compute against the hand-labeled ground truth: **true positives** (flagged AND labeled S), **false positives** (flagged AND labeled B or T), **false negatives** (not flagged AND labeled S), **true negatives** (not flagged AND labeled B or T). Compute precision = TP / (TP + FP), recall = TP / (TP + FN), and F1 = 2 × precision × recall / (precision + recall). Report a per-detector results table with these five metrics plus raw counts of S-recall, B-flagged, T-flagged. **Part 4 — combination tests.** Test three combinations against the same ground truth. Combination 1 — A OR B (phrase-grep OR tool-failure-bracketed): flag if either detector flags. Combination 2 — B OR C (tool-failure-bracketed OR write-tool-adjacent): flag if either flags. Combination 3 — (A OR B) AND C (must be both a "decision-y" event and near a write operation). Report the same precision/recall/F1 table for each combination. **Part 5 — qualitative analysis.** For the single best-performing detector or combination by F1, identify the S-class blocks it MISSED (false negatives). Quote one to three representative misses and describe the common pattern — is it a phrase the grep list doesn't catch, a decision not adjacent to a tool call, a decision in a short narration block, etc.? For the same detector or combination, identify the T-class blocks it FALSELY FLAGGED (false positives). Quote one to three representative false flags and describe the common pattern. The objective is to make the failure modes legible so the CEO can judge whether the precision/recall numbers reflect tolerable failure modes or unacceptable ones. **Part 6 — implementation-shape sketches.** For the winning detector or combination by F1, sketch both implementation paths at one-paragraph depth each. **Bellows-side path:** what would change in `bellows/runner.py` or `bellows/verdict.py` to surface flagged blocks into the verdict request file as a new section? Estimate LOC (production + tests), call sites touched, risk surfaces (parser-touching code per v4.38 Restart Discipline, gate behavior changes, contract changes to the verdict request schema). **Planner-side path:** what would Rule 22 need to specify, what tooling would the Planner need, estimated added cognitive cost per Rule 22 verification, and risk of inconsistent execution. Do NOT recommend one over the other. Present the two paths' tradeoffs and let the CEO choose. Do NOT propose a fix plan. Disposition is methodology selection and implementation-path-shape only. Deposit findings to `bellows/knowledge/research/intermediate-decision-detection-design-2026-05-12.md` with this structure: header (plans audited, block counts, label distribution), Part 1+2 hand-classification summary, Part 3 four-detector results table, Part 4 three-combination results table, Part 5 best-detector qualitative failure analysis with quoted examples, Part 6 Bellows-side and Planner-side implementation sketches with LOC and risk surfaces side-by-side.
