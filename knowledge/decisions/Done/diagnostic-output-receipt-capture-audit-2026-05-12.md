# Diagnostic — Output Receipt Capture Completeness Audit

**Execution Map:** Step 1 (BELLOWS_SYSTEMS_ANALYST)

**Purpose:** Answer one question empirically. **Does Bellows's Output Receipt successfully capture every decision the agent surfaces during `claude -p` execution?** The concern: interactive Claude Code sessions surface tool prompts, file-access prompts, and "this looks off, proceed?" pauses in real time. `claude -p` headless mode collapses those into agent self-decisions that the agent narrates somewhere in the stdout stream. If those narrated decisions don't make it into the structured Output Receipt that the Planner inspects, the headless workflow has a structural visibility gap regardless of agent discipline. This audit is a mechanism check — given the words the agent produces, is the receipt capture lossless or lossy? Out of scope: whether the agent should have surfaced decisions they didn't (agent-discipline question), whether the Planner reads the receipt during Rule 22 (Planner-workflow question), whether the receipt's summary depth is sufficient (format question). Population: 14-day window (2026-04-28 → 2026-05-12) across Bellows and invoice-pulse Done/ plans. Read-only, classification only, no recommendations.

State the partition in the findings header. **Part 3 — decision marker grep in raw stdout.** For each plan in scope, grep every step's raw `claude -p` stdout artifact for decision-indicating phrases. Starting grep target list: `had to`, `decided to`, `on initiative`, `noticed that`, `chose to`, `I'll update`, `I need to update`, `I'll modify`, `not in the plan`, `not mentioned`, `unclear from the plan`, `assumed`, `defaulted to`, `since the plan`, `the plan doesn't specify`, `went with`, `opted for`, `corrected the`, `I'll go with`, `let me adjust`. Run case-insensitive. Tally hits per plan per step. If during this part you encounter recurring decision-surfacing phrasing not in the list (read 3-5 of the higher-hit stdout files for context), extend the grep set and re-run for consistency. Report the final extended grep list at the top of the findings file. Report total stdout-detected decision instances across the sample. **Part 4 — decision marker grep in Output Receipt.** Run the SAME extended grep set against the structured Output Receipt artifact for the SAME steps. Tally hits per plan per step. Report total Output-Receipt-detected decision instances. **Part 5 — instance-level cross-reference.** For each decision-marker hit found in Part 3 (stdout), check whether the same instance appears in Part 4 (Output Receipt). Same-instance matching is conservative: a phrase in stdout and a phrase in the Output Receipt are the SAME instance if they refer to the same agent decision in the same step. Two separate sentences in stdout that both happen to contain `decided to` are two separate instances. The Output Receipt may rephrase or summarize — accept a paraphrased match (same decision content, different words) but require the underlying decision to be identifiable. Classify each stdout instance as: **(α) captured** — instance appears in stdout AND is identifiable in the Output Receipt (verbatim, paraphrased, or summarized). **(β) leak** — instance appears in stdout but is NOT identifiable in the Output Receipt. For each β instance, quote the stdout sentence (with surrounding context, ~2 sentences before and after) and quote the closest Output-Receipt section that you would have expected to contain it, so the CEO can judge whether the absence is meaningful (a real decision that vanished) or trivial (a discarded verbal aside the Output Receipt correctly summarized away). **Part 6 — capture-completeness summary.** Compute and report: total stdout-detected instances, total α, total β, capture rate (α / total stdout instances), per-project capture rates, worst three plans by β count (highest-leak plans with absolute leak count and percentage), best three plans by capture rate (sanity baseline). If the capture rate is ≥95%, the mechanism is effectively lossless and any decision-visibility concern is downstream of Bellows. If the rate is 80-95%, the mechanism has measurable leakage that warrants follow-up. If <80%, the mechanism is structurally lossy. **Part 7 — representative β instances.** Select 3-5 β instances from across the sample that best illustrate the leak pattern. For each, present a side-by-side: stdout context (verbatim, ~3 sentences) vs. corresponding Output Receipt section (verbatim, the part of the receipt that should have captured the decision). The selection should span: at least one β where the leaked decision looks substantive (file modification, scope expansion, test fixture change), at least one where it looks borderline (clarification choice, default assumption), at least one where it looks trivial (verbal aside the receipt correctly omitted).
