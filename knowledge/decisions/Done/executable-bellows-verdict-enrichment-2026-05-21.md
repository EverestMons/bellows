The Planner currently performs Rule 22 (a)+(c)+(d) mechanical checks on every deposited plan — opening verdict request, QA report, and dev log per close. This plan moves the mechanical reads into Bellows, surfaces results in an enriched verdict request, and keeps the Planner in the close loop with one file to read instead of three. Auto-continue / auto-close on all-green is explicitly out of scope — every close still gets Planner review.

The Rule 25 routing row for `rule_22_check_failed` shipped 2026-05-21 in PLANNER_TEMPLATE v4.46 — Planner-side routing is already in place.

**Restart Discipline (per 2026-05-11 LESSONS):** This plan adds `_gate_rule_22_verification`, which is a gate that THIS plan's QA step would run if the daemon had hot-reloaded. Bellows has no hot-reload; the running daemon executes pre-fix code through this plan's lifecycle. QA in this plan will NOT see the new gate fire because the daemon code is stale relative to the DEV commit. This is structurally expected, not a defect. After this plan closes, the daemon must be restarted to load the new gate code; subsequent plans dispatched after restart will see the enriched verdict requests.
