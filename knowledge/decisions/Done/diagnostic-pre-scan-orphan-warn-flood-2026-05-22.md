# Bellows — pre-scan orphan verdict-* WARN flood
**Date:** 2026-05-22 | **Tier:** Diagnostic | **Dispatch Mode:** bellows | **Test Scope:** targeted | **Execution:** Step 1 (SA) | **pause_for_verdict:** after_step_1 | **auto_close:** false

## Context

`bellows/knowledge/BACKLOG.md` open item (added 2026-05-21): the pre-scan introduced by `executable-consume-verdicts-prefix-fix-2026-05-21` unconditionally renames every `processed-verdict-*.md` file in `bellows/verdicts/resolved/` back to canonical `verdict-*.md` so the main filter loop in `_consume_verdicts` can re-evaluate them. The fix is correct for active plans where the originating `verdict-pending-*` plan still exists in a watched `decisions/` directory. The side effect: when the originating plan has long since moved to `Done/` (or `halted-`), the main-loop no-match branch fires (`no verdict-pending plan found step N — leaving in resolved/ for retry`) every 30s for each orphan, producing a WARN flood. Eight files were observed in this state after the 21:31:34 mass-rename event on 2026-05-21.

CEO has selected resolution shape (1) from the BACKLOG entry: skip rename when no paired `verdict-pending-*` plan exists in any watched decisions directory. This preserves the bug-fix for active plans and suppresses orphan creation at the source. Shape (2) — detect-and-revert in the main-loop no-match branch — was rejected as symptom-masking.

**Scope:** read-only investigation of `bellows/bellows.py`, `bellows/gates.py`, and `bellows/verdict.py` plus filesystem inspection of `bellows/verdicts/resolved/` and the watched `decisions/` directories. No fixes, no code changes. Findings deposit goes to `bellows/knowledge/research/`. The Planner reads findings and authors a separate executable.
