## Context

Three reliability bugs observed during the 2026-04-14 Anvil session:

**Bug A — Plan completion detection fires before closing steps verify.** Bellows marked `executable-anvil-cycle-11-2026-04-14.md` as ✅ DONE and sent queue-drain notification, but the plan's own final step (move file from `in-progress-` to `Done/`) never executed. The file stayed at `in-progress-` on disk. Bellows trusted the agent's successful return code without confirming the plan's own closing operations actually ran.

**Bug B — Same file detected twice.** When the noise-reduction recovery plan was deposited, Bellows logged it as detected and running TWICE: `Bellows: detected plan executable-qa-recovery... ▶ started ... detected plan executable-qa-recovery... ▶ started`. Two `✅ DONE` lines followed, first at $0.4480, second at $0.00 (no-op). The `_seen` set did not prevent the double-pickup. Possible race between the file rename to `in-progress-` and the filesystem watcher re-firing for the new name.

**Bug C — Code changes require process restart (no hot-reload).** The `is_runnable_plan` parallel-prefix fix shipped cleanly (commit, tests, QA) earlier in the same session. But the running Bellows process loaded `is_runnable_plan` at module-import time when it was started at 1:12PM. After the fix committed, two `parallel-1-diagnostic-*.md` files were deposited into watched decisions folders — Bellows' log shows zero detection events for either file. The running process is still executing the old `is_runnable_plan` that rejects `parallel-` prefixes. **Every Bellows bug fix has this property:** the fix exists on disk but does not take effect until someone manually kills and restarts the Bellows process. There is no hot-reload, no version check on startup against git HEAD, no warning when the running code diverges from disk.

All three are reliability issues, not correctness issues — none corrupts data. But Bug C is a meta-bug: it means every OTHER Bellows fix this diagnostic might recommend has the same deployment gap. This diagnostic investigates root causes for A, B, and C.
