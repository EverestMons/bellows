## Context

Three documented false-positive fires against the GUARDRAILS-prescribed `rm -f .git/index.lock …` pattern (2026-04-17, 2026-05-06, 2026-05-21). Zero true-positive `.git/` denial fires on record. The denial itself originates from Claude Code's runtime, not bellows code — this plan does NOT attempt to suppress the denial, only to prevent bellows from converting it to `gate_failure`. Architecturally parallel to the existing `READ_CLASS_TOOLS` exemption at line 36.

## Recursion-risk constraint (LESSONS 2026-05-17, 2026-05-19)

The DEV agent's own git operations during this plan MUST NOT use the GUARDRAILS-prescribed `rm -f .git/index.lock …` recovery — running it would itself trip the very bug being fixed and produce a recursive gate_failure. The DEV step uses ordinary git commands (`git add`, `git --no-pager commit`, `git --no-pager push`) without lock-cleanup preamble. If a true orphaned lock is encountered, the agent halts and reports to the CEO rather than improvising recovery.
