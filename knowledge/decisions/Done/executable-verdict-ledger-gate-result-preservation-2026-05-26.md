## CEO Context

The verdict ledger has been writing empty arrays for `gate_failures` and `files_changed` on the three `_consume_verdicts` write paths since the consume-time empty-dict construction was introduced. The real `gate_result` is captured at gate-evaluation time and rendered into the pending verdict request's markdown, then destroyed when `_consume_verdicts` constructs a fresh empty dict at `bellows.py:1226`. Only the auto-close path at `bellows.py:656` writes real ledger data today. Every CEO-verdict close path (continue-to-done, continue-resume, stop/halt) loses the gate evidence by the time `log_to_ledger` is called.

The SA diagnostic at `bellows/knowledge/research/verdict-ledger-gate-result-preservation-2026-05-26.md` evaluated four shapes (E.1 markdown parser, E.2 structured sidecar, E.3 write-at-post-time, E.4 JSON metadata line) and identified E.4 as unconditionally dominant: ~7 LOC, zero new artifacts, zero coupling (JSON is self-describing), and automatic backward compatibility (pre-fix pending files lack the metadata line → init-to-None → falls through to existing empty-dict default = today's behavior).

Fix F is complementary: the two terminal log sites at `bellows.py:495` and `bellows.py:586` currently log only failure count. Expanding to include gate names and `files_changed` count gives real-time triage signal that the ledger fix alone doesn't provide for live debugging.

Full-suite scope is mandated under Rule 21: `_consume_verdicts` is load-bearing daemon infrastructure (session management equivalent for plan state), and the SA's call-flow trace identified four test files that reach the affected code paths (`test_consume_verdicts.py`, `test_misplaced_verdicts.py`, `test_bellows.py`, `test_verdict.py`).

Total estimated change: +9 LOC across `verdict.py` and `bellows.py`. The Q6 hand-off in the diagnostic contains exact line numbers and verbatim code snippets.
