## Section 2 — Current Write→Merge Mechanics

### Trace: One Ledger File Through the Conflict Path

Using `knowledge/research/agent-prompt-feedback.md` as the exemplar (highest write frequency):

1. **Agent writes in worktree.** The agent (running inside `wt_path = project_path/.bellows-worktrees/<slug>`) reads the current `agent-prompt-feedback.md`, appends a dated entry, and optionally updates the Patterns Identified section. The agent runs `git add knowledge/research/agent-prompt-feedback.md` and `git commit -m "docs: prompt feedback — [agent] [brief description]"` inside the worktree.

2. **Step completes.** `runner.run_step()` returns parsed output to `bellows.py:run_plan()`.

3. **Auto-stage deposits** (bellows.py:1003–1071, `_auto_stage_deposits`). Declared deposits are git-added and committed in the worktree if unstaged. `agent-prompt-feedback.md` is NOT a declared deposit (it's not in the `**Deposits:**` block) — it relies on the agent's own commit. However, it IS in `SCOPE_ALLOWLIST` (gates.py:25), so scope_check doesn't reject it.

4. **Gates check** (bellows.py:580, `gates.check()`). The file appears in `files_changed` via `_parse_diff_stat`. Scope_check passes because the file is in `SCOPE_ALLOWLIST`.

5. **Pause/close decision** (bellows.py:602–640 for non-final, 711–750 for final step).

6. **Teardown** (bellows.py:622 or 731–732). `_teardown_worktree(project_path, wt_path, plan_slug)` is called.

7. **Merge — primary (ff-only)** (bellows.py:1190–1193). `git merge --ff-only <branch_name>` with `cwd=project_path`. If `main` has not advanced since the worktree was forked (no other plan merged first), this succeeds — the branch pointer moves forward, and the worktree's commit (including the `agent-prompt-feedback.md` append) lands on main.

8. **Merge — fallback (no-ff)** (bellows.py:1196–1200). If another plan already merged (advancing `main`), ff-only fails. The `--no-ff` fallback runs a 3-way merge. **This is where the conflict occurs**: both the merged plan and this plan appended content at EOF of `agent-prompt-feedback.md`. Git's 3-way merge sees the common ancestor (pre-append state), left (first plan's append), right (this plan's append) — both modified the same region (EOF), producing a conflict.

9. **Conflict → abort → raise** (bellows.py:1201–1209). `git merge --abort` runs. `WorktreeTeardownError` is raised. The plan transitions to `verdict-pending-*` with `gate_failure` pause reason and a `worktree_teardown` gate failure entry.

### The Daemon's Relationship to `main`

The daemon process (`bellows.py`) runs from the **canonical project root** (`project_path`), which is the real git checkout on `main`. The daemon's own writes are:

- **lifecycle.db** — SQLite file, not git-tracked (in `.gitignore`)
- **Verdict files** — `verdicts/pending/*.md`, `verdicts/resolved/*.md` — daemon-owned paths
- **Plan renames** — `shutil.move` of plan files between lifecycle prefixes (`in-progress-`, `verdict-pending-`, `halted-`, `Done/`)
- **Log files** — `logs/*.json`, not git-tracked
- **Shadow cache** — `.bellows-cache/` contents, not git-tracked

**None of these daemon writes touch the shared ledger files.** The daemon NEVER writes to `PROJECT_STATUS.md`, `agent-prompt-feedback.md`, or `FORWARD.md`.

### The Conflict-Free Writer Property

**Claim:** A writer that commits directly to `main` in `project_path` (without going through a worktree branch + merge) is conflict-free.

**Proof:** A conflict requires two divergent branches modifying the same region of the same file. When the daemon writes directly to the `project_path` working tree (which IS `main`), it performs:
1. Write file content → `git add <file>` → `git commit` — all with `cwd=project_path`
2. This creates a new commit directly on `main`. No branch divergence, no merge, no conflict.

**Serialization:** Even if two plan threads complete near-simultaneously, their post-merge daemon writes would serialize via git's `index.lock` mechanism — the second `git add` waits for the first `git commit` to release the lock. The daemon is a **serialized sole writer** on `main` for any file it writes post-merge.

**This is the property the design exploits:** move ledger writes from agents-inside-worktrees (which merge and conflict) to the daemon-on-main (which writes directly and serializes).

## Section 3 — Relocation Design (Per-File Disposition)

### File 1: `PROJECT_STATUS.md`

**Current:** QA agent appends a milestone entry inside the worktree, commits, worktree teardown merges to main.

**Recommended disposition: (a) daemon-written post-merge.**

- The QA agent produces the milestone summary text as part of its Output Receipt (the `### What Was Done` section already contains this information).
- The agent does NOT write to `PROJECT_STATUS.md` inside the worktree.
- After successful code merge, the daemon extracts the milestone text from the agent's output (or from a dedicated structured field — see Section 4) and appends it to `PROJECT_STATUS.md` on main, then commits.

**Why not (b) DB-stored + generated?** PROJECT_STATUS entries are rich narratives — agent-written prose with cross-references to dev logs, QA reports, commit SHAs, and plan IDs. The lifecycle DB captures structured metadata (lifecycle_state, closed_at, cost, duration) but not the narrative synthesis. Generating equivalent prose from structured data would require an LLM call at generation time, which conflicts with the "thin Layer 1 infrastructure" principle. The daemon-post-merge path preserves the agent's narrative while moving the write out of the merge path.

**Trade-off:** The agent loses the ability to read the current `PROJECT_STATUS.md` and position its entry relative to existing content. Mitigation: the daemon always appends at a canonical position (e.g., after the first `## Completed` line), so positioning is deterministic.

### File 2: `knowledge/research/agent-prompt-feedback.md`

**Current:** Every agent at every step reads the file, appends a dated entry, optionally updates the Patterns Identified section, and commits in the worktree.

**Recommended disposition: (b) DB-stored + generated.**

- Feedback entries are structurally regular: date, agent name, 5 bullet-point categories.
- The lifecycle DB can store feedback as rows in a new `prompt_feedback` table: `(step_id, agent, plan_slug, dated_entry_text)`.
- The agent emits the feedback text via the Output Receipt or a structured channel (Section 4). The daemon writes it to the DB post-step.
- The `agent-prompt-feedback.md` file becomes a generated report (like the forge reconstruction reports): `SELECT * FROM prompt_feedback WHERE project = ? ORDER BY created_at DESC`.

**Why (b) over (a)?** The file is already 309KB and growing. Every step of every plan appends to it. This is the highest-frequency conflict source. DB storage eliminates the append-at-EOF collision entirely. The Patterns Identified section can become a separate Planner-maintained artifact (session-wrap synthesis) rather than a live-edited section inside an unbounded append log.

**Trade-off:** The "read then extend Patterns" workflow changes. Currently, agents read the Patterns section and add new patterns. Under DB-generated model, the Patterns section becomes Planner-maintained at session-wrap (reading DB entries and synthesizing). This is acceptable because the Planner already reads agent-prompt-feedback.md at session-wrap (PLANNER_TEMPLATE line 1504).

### File 3: `knowledge/FORWARD.md`

**Current:** Agents may add new rows (filing deferred work). Planner does status-update edits at session-wrap (already on main, not in worktree).

**Recommended disposition: (a) daemon-written post-merge for new-row additions.**

- Agent emits new FORWARD entries via the Output Receipt `### Flags for Next Step` section or a dedicated structured channel (Section 4).
- The daemon appends new rows to FORWARD.md on main post-merge.
- Planner-direct session-wrap edits (row status updates) are unchanged — they're already on main.

**Why not (b)?** FORWARD.md is a lightweight standing queue with 22 rows. It's human-curated, hand-reconciled at session-wrap (Rule 42), and directly read by the CEO. DB-generating it would add complexity without proportionate benefit. The file is small and stable; only new-row additions from agents cause conflict.

**Why not (c) leave in worktree?** New-row additions from parallel plans collide at EOF — the same structural conflict as PROJECT_STATUS and agent-prompt-feedback.

### File 4: `knowledge/BACKLOG-ARCHIVE.md`

**Recommended disposition: (c) leave as-is — no action needed.**

Frozen read-only since 2026-06-12 (Reporting Phase 3 cutover). No writes, no conflicts.

### File 5: Governance-root `LESSONS.md`

**Recommended disposition: (c) leave as-is — out of scope.**

LESSONS.md is at the governance root (`/Users/marklehn/Developer/GitHub/LESSONS.md`), outside the bellows project git tree. It does NOT participate in `_teardown_worktree` merges. Agent writes go directly to the governance-root working tree, not through a bellows worktree branch. Concurrent-write races (agent vs. Planner on the governance-root checkout) are a separate issue from the worktree merge conflict class addressed here.

### Disposition Summary

| File | Disposition | Mechanism |
|---|---|---|
| `PROJECT_STATUS.md` | **(a) daemon-post-merge** | Agent emits text via channel; daemon appends on main after code merge |
| `agent-prompt-feedback.md` | **(b) DB-stored + generated** | Agent emits text via channel; daemon writes DB row; .md is generated |
| `FORWARD.md` | **(a) daemon-post-merge** (new rows only) | Agent emits new rows via channel; daemon appends on main after code merge |
| `BACKLOG-ARCHIVE.md` | **(c) leave as-is** | Frozen |
| `LESSONS.md` | **(c) leave as-is** | Out of scope (different repo) |

## Section 4 — The Agent→Daemon Content Channel

### The Problem

Under the new design, agents must communicate ledger content to the daemon WITHOUT writing the shared file inside their worktree. The daemon applies the content to main post-merge. How does the content travel from agent to daemon?

### Option A: Structured Field in the Output Receipt

**Mechanism:** Extend the Output Receipt format (BELLOWS_SYSTEMS_ANALYST.md lines 76–102 and all specialist files) with new sections:

```markdown
### Ledger Updates
#### PROJECT_STATUS
- 2026-06-13: **[Plan title].** [Milestone summary text]

#### Prompt Feedback
[Full dated feedback entry]

#### FORWARD Additions
| # | Added | Item | Type | Plan-id link | Status |
|---|---|---|---|---|---|
| 23 | 2026-06-13 | [Item text] | deferred-work | 42 | open |
```

The daemon's parser (`parser.py`) already extracts `ceo_flags` from `### Flags for CEO` in the Output Receipt (parser.py:30–37). Extending it to extract `### Ledger Updates` is a natural fit.

**Integration points:**
- `parser.py:6` (`def parse`) — add extraction of `### Ledger Updates` subsections
- `bellows.py:~622/731` — after successful `_teardown_worktree()`, call a new `_apply_ledger_updates(parsed, project_path, plan_id)` function
- `_apply_ledger_updates` writes to `PROJECT_STATUS.md`, the prompt_feedback DB table, and `FORWARD.md` on main, then `git add` + `git commit`

**Ordering/serialization:** Daemon writes are serialized by git's `index.lock`. For parallel plans completing near-simultaneously, the second plan's ledger write blocks on the lock until the first commits. Ordering key: merge completion time (naturally serialized by git). This is deterministic and needs no explicit serialization mechanism.

**Idempotency:** The daemon records ledger writes in `lifecycle.db` (new table `ledger_writes`: `step_id, ledger_file, content_hash, applied_at`). On retry/resume, the daemon checks the hash before re-applying. Duplicate detection prevents double-append.

**Pros:** Uses existing infrastructure (Output Receipt, parser). Agent output is already captured and available at teardown time. No new files or directories.

**Cons:** Agent output text can be large and noisy — parsing structured content from Markdown is fragile. If the agent doesn't produce the expected format, the ledger update is silently lost (unless the daemon gates on it).

### Option B: Per-Step Deposit File in a Non-Merged Location

**Mechanism:** The agent writes ledger content to a structured file at a path the daemon reads but that is NOT merged into main:

```
.bellows-cache/ledger-updates/<plan-id>-step-<N>.json
```

The `.bellows-cache/` directory is gitignored and lives in the canonical project root (not inside the worktree). The agent writes to it via absolute path:

```json
{
  "plan_id": 42,
  "step": 1,
  "project_status_entry": "...",
  "prompt_feedback_entry": "...",
  "forward_additions": [...]
}
```

After successful code merge, the daemon reads the JSON and applies the updates.

**Integration points:**
- Agent prompt instructions: "Write ledger updates to `<project_path>/.bellows-cache/ledger-updates/<plan-id>-step-<N>.json` (NOT to the ledger files themselves)"
- `bellows.py:~622/731` — after `_teardown_worktree()`, read and apply the JSON
- Cleanup: daemon deletes the JSON after successful application

**Ordering/serialization:** Same as Option A — git index.lock serializes concurrent writes.

**Idempotency:** The JSON file's existence is the "not yet applied" signal. After application, the daemon deletes it. On crash-restart, undeleted JSONs are re-applied. Content-hash dedup in lifecycle.db prevents double-append.

**Pros:** Structured JSON is more robust to parse than Markdown. The content is persisted on disk before teardown, surviving daemon crashes. Clear separation between agent output (conversational) and ledger content (structured).

**Cons:** Requires agents to write to a path OUTSIDE their worktree (the canonical `.bellows-cache/`). The agent's `claude -p` process runs with `cwd=wt_path` — writing to an absolute path outside the worktree is architecturally unusual but mechanically sound (the agent has filesystem access). Adds a new file format and lifecycle to manage.

### Option C: Extension of the Existing Deposits/Gate System

**Mechanism:** Treat ledger updates as a special class of deposit. The agent writes a "ledger deposit" file in the worktree at a known path (e.g., `.bellows-ledger-updates/step-<N>.json`). The daemon's `_auto_stage_deposits` stages it. After teardown merge lands it on main, the daemon reads and applies the content to the actual ledger files, then deletes the intermediary.

**Integration points:**
- New convention: `.bellows-ledger-updates/` directory in worktree for structured ledger content
- `_auto_stage_deposits` already handles declared deposits — extend to include ledger deposits
- Post-merge: daemon reads `.bellows-ledger-updates/step-<N>.json` from main, applies to ledger files, commits, deletes intermediary

**Ordering/serialization:** The intermediary merges cleanly (each plan writes to its own file — no collision). The application step is daemon-on-main, serialized by index.lock.

**Idempotency:** Intermediary file existence = not yet applied. Deletion = applied.

**Pros:** Stays within the existing worktree/merge model. Each plan's ledger deposit is its own file (no collision). The intermediary merge is safe because file names are unique per plan.

**Cons:** Two-phase write: agent writes intermediary, daemon reads and applies. Adds complexity to the merge path (more files to merge, more post-merge cleanup). The intermediary directory accumulates on main if the daemon crashes between merge and cleanup.

### Recommendation: **Option A (structured Output Receipt field)**

**Rationale:**

1. **Minimal infrastructure change.** The parser already extracts structured content from the Output Receipt. Adding `### Ledger Updates` is one regex extraction per subsection.

2. **No new files or directories.** No `.bellows-cache/ledger-updates/` to manage. No `.bellows-ledger-updates/` intermediary.

3. **Natural fit.** The Output Receipt is the established channel for step-to-daemon communication. `### What Was Done` → project status. `### Flags for CEO` → escalation. `### Ledger Updates` → ledger content.

4. **Failure mode is clean.** If the agent doesn't produce the section, no ledger update is applied. This is a visible omission (the daemon can log "no ledger updates found") rather than a silent data loss.

**Decision fork for CEO:** Option A vs. Option B. Option A is simpler but depends on agent output format compliance. Option B is more robust but adds filesystem complexity. Option C adds complexity to the merge path (counterproductive — the goal is to simplify merges).

## Section 6 — Failure Modes & Migration

### Failure Semantics Comparison

| Scenario | TODAY | UNDER NEW DESIGN |
|---|---|---|
| Two parallel plans both append to agent-prompt-feedback.md | **Merge conflict at teardown of the second plan.** `WorktreeTeardownError` raised. Plan strands at `verdict-pending-*`. Entire plan blocked — code merge AND ledger update both fail. Requires manual R2 recovery. | **Code merges succeed** (no shared-file collision). Daemon applies ledger updates sequentially post-merge. Both plans' code AND ledger content land. **No conflict.** |
| Daemon crashes between code-merge and ledger-write | N/A — today the ledger is part of the code merge. | **Code lands but ledger update is lost.** Recoverability: (1) The agent's output is logged to `logs/<slug>-step-<N>.json` — the `result_text` field contains the Output Receipt with ledger content. The daemon can replay from logs on restart. (2) The lifecycle DB records which steps completed — a startup recovery scan can identify steps with no `ledger_writes` record and replay from logs. **Assessment: recoverable.** |
| Agent doesn't produce the Output Receipt `### Ledger Updates` section | N/A — today the agent writes directly. | **Ledger update is silently omitted.** Mitigation: the daemon logs "no ledger updates found for step N" — visible in daemon logs. A new informational gate (`ledger_updates_present`) could WARN but not block. This is acceptable because omission is a prompt compliance issue, not a data loss — the content was never produced, not produced-and-lost. |
| Agent produces malformed Ledger Updates content | N/A — today the agent writes directly (malformed content is the agent's problem). | **Daemon parse fails, ledger update lost.** Mitigation: daemon logs the parse error with the raw content. Manual recovery: read the log, apply manually. This is BETTER than today's merge conflict — a parse error doesn't strand the plan. |

### Is Today's Failure Mode Acceptable vs. the New Design's?

**Today:** A ledger conflict strands the ENTIRE PLAN at teardown. Code changes are not lost (they're on the worktree branch), but they're not landed either. Recovery requires manual R2 intervention (merge abort, manual conflict resolution, re-merge). This blocks the entire pipeline for the affected project.

**New design:** If the daemon crashes between code-merge and ledger-write, the code lands but the ledger update is lost. This is **strictly better** because:
1. Code is not blocked by a ledger issue — the primary deliverable lands.
2. Ledger content is recoverable from logs (the agent's output is always captured to `logs/`).
3. The loss mode is a daemon crash (rare) rather than a structural merge conflict (systematic for parallel plans).

**Assessment: the new design's failure mode is acceptable and strictly better than today's.**

### Migration Strategy

**Recommended: Phased migration.**

#### Phase 1: agent-prompt-feedback.md (highest frequency, lowest risk)

1. **Daemon-side:** Add `_apply_ledger_updates()` function to bellows.py. Add `prompt_feedback` table to lifecycle.db. Add `### Ledger Updates > #### Prompt Feedback` extraction to parser.py.
2. **Governance:** Update PLANNER_TEMPLATE feedback protocol (3 sites: lines 265–274, 308–317, 341) to instruct agents to emit feedback via Output Receipt instead of writing the file directly.
3. **Coexistence:** During transition, the daemon checks: if `agent-prompt-feedback.md` was modified in the worktree (visible in `files_changed`), skip the daemon-side write (agent used the old protocol). If not, apply the Output Receipt content. This allows old-format plans (in-progress at transition time) to complete without breakage.
4. **Verify:** Run 3–5 plans under the new protocol. Confirm no merge conflicts on agent-prompt-feedback.md. Confirm feedback entries appear in the DB and/or generated file.
5. **Cut over:** Remove agent-prompt-feedback.md from `SCOPE_ALLOWLIST`. Any plan that still writes directly will trip `scope_check` — this is the enforcement mechanism.

#### Phase 2: PROJECT_STATUS.md (high frequency, moderate risk)

Same pattern as Phase 1:
1. Add `### Ledger Updates > #### PROJECT_STATUS` extraction.
2. Update Rule 8 in PLANNER_TEMPLATE.
3. Coexistence check on `files_changed`.
4. Verify with 3–5 plans.
5. Remove from `SCOPE_ALLOWLIST`.

#### Phase 3: FORWARD.md new-row additions (low frequency, low risk)

1. Add `### Ledger Updates > #### FORWARD Additions` extraction.
2. Update PLANNER_TEMPLATE Rule 42 note and any step templates that file FORWARD rows.
3. Verify.

#### Why Phased Over Big-Bang?

1. **Coexistence window.** Old-format plans that are in-progress at the time of each phase's governance edit must complete under the old protocol. The coexistence check (detect whether the agent wrote the file directly) handles this gracefully.
2. **Rollback granularity.** If Phase 1 reveals an unexpected issue (e.g., agent output format compliance is lower than expected), Phases 2–3 can be deferred without losing Phase 1's benefit.
3. **Risk scaling.** agent-prompt-feedback.md is the highest-conflict file and the lowest-risk migration (the content is a log, not a status artifact). PROJECT_STATUS.md is a higher-stakes artifact (CEO reads it directly). Phasing lets Phase 1 validate the mechanism before Phase 2 stakes more on it.
